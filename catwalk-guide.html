01

Product rationale

Catwalk solves a specific, real problem: neighbourhood cats are invisible to the people who love them. You might walk past the same black cat on Brownlow Road every morning for two years and never learn its name, who feeds it, or whether anyone else knows it exists.

The product is a community-maintained geographic directory of cats — not a social network, not a photo app. The map is the product. Everything else (profiles, visits, contributions, slow blinks) is infrastructure that makes the map richer over time.

Core thesis: Street-level community knowledge should be shareable and persistent. The person who moves away shouldn't take the cat's name with them. Catwalk is the record.

The product is designed with two types of user in mind: the contributor (someone who adds and enriches cat profiles) and the walker (someone who wants to find cats in a new area). Both are served by the same data model — one writes, the other reads — and the friction to switch between roles is intentionally low.

Core user flows

Five primary flows cover all meaningful user activity in the app.

Flow A — First visit

1

Landing + location consent

User lands on the login wall. Location consent modal fires on first load. The anti-theft statement is visible before the user has any access to data.

2

Registration

Email + password. Optional profile photo (gallery or live camera). Identity type: Human of a cat or Unattached catwalker. Location string (free text, not precise coordinates).

3

Map loads centred on user

Geolocation resolves, map flies to user position. Cat emoji markers appear. "1 cats in this area" counter top-right. Location saved to localStorage for subsequent sessions.

Flow B — Adding a cat

1

Pin placement

Tap "Drop pin" or tap directly on the map. Draggable marker. Location auto-resolves to area + city + country via Nominatim reverse geocoding.

2

Duplicate detection

Within a 200m radius, existing cats are checked. If found, the user is shown potential matches and can attach their photo to an existing profile rather than create a duplicate.

3

Cat form

Name, emoji (colour-coded: 8 types), description, personality traits (multi-select), optional photo. Address field pre-filled from reverse geocode, editable.

4

Saved to Firestore

Cat document created with creator attribution, contributor array, serverTimestamp(). User's catsFound counter incremented. Map marker appears immediately via onSnapshot listener.

Flow C — Visiting a cat profile

1

Open profile

Tap map marker or browse card. Full-screen profile slides up. Photos, description, characteristics, blurred location map, community descriptions, contributors.

2

Mark as Visited

Records a visit with timestamp. 24-hour cooldown enforced per cat to prevent systematic data collection. Total visit count and personal visit count displayed separately.

3

Slow Blink

Cats' traditional greeting. Recorded as a community affection metric. No limit. Count visible on profile.

Flow D — Contributing to a cat

1

Open Contribute form

Editable fields: description, personality traits, alternative names, behaviour characteristics. Plus a community description section with type selector (description / anecdote / behaviour).

2

Community descriptions

Each description saved as a separate object in the cat's descriptions[] array via arrayUnion. Displays below the blurred map with type badge, date, and first name of contributor.

Flow E — Catspotting

1

Upload or capture photo

Dedicated tab for quick in-the-moment spotting. Photo capture via gallery or live camera. Attempts GPS extraction from EXIF data to find nearby cats automatically.

2

Match or create

If nearby cats found within 500m: user prompted to confirm which cat. If no match: option to create a new profile from the catspotting photo.

03

Component architecture

The entire application is a single App.tsx file. This was a deliberate constraint during rapid iteration — all state lives in the root CatwalkApp component and is passed down via props. There are no external state management libraries, no context providers, no routing library.

CatwalkApp

Root component. Holds all global state: auth, cats array, filters, selected cat, modal visibility flags. Firebase listeners live here.

HeaderBar

Fixed top bar, 56px. Clickable Catwalk title returns to map. Circular avatar (profile photo if set) opens a dropdown sheet with name, contribution count, "My profile", and "Sign out" — designed for mobile tap targets. "How to use" link opens the guide.

CatList (Browse)

Title bar + Add Cat button at top, then search (name, area, city, country, alternative names) and filter row below. Inline country/city filter chips derived from live data. Filter modal for trait/behaviour filters. All stackable.

CatProfile

Full-screen cat detail. Photos with focus picker, visit/blink actions, blurred map, characteristics, community descriptions, contributor attribution with clickable public profiles.

MapSnapshot

Reusable Leaflet mini-map. In blur mode: pin randomised ±500m, zoom pulled back 3 levels, CSS blur applied. Protects exact location without removing neighbourhood context.

AddCatForm

Multi-step cat creation. Location from GPS, manual pin drag, or search. Duplicate detection within 200m. Reverse geocoding via Nominatim.

ContributeForm

Edit a cat's known information. Fields include description, personality traits, living situation, alternative names, plus community description with type selector.

CatspottingScreen

Quick-capture flow. Photo → EXIF location extraction → nearby cat matching → attach or create. Centred empty state, live camera option.

UserProfile

Own profile: stats grid (clickable), edit form with live photo overlay, granular contribution management. Delete sheet shows individual photos, visits, and descriptions as checkboxes. Cats can be deleted entirely or anonymised independently.

PublicUserProfileModal

Another user's profile. Stats grid with "Cats added" as a tappable trigger — opens a bottom sheet listing that user's cats, each linking to the cat profile.

PhotoCaptureButton

Reusable wrapper component. Presents gallery vs. live camera choice. Handles getUserMedia stream, canvas capture, and graceful fallback to file input.

FilterModal + FilterState

Trait/behaviour/petting filters. FilterState extended to include city and country. Country selection auto-narrows city options. All filters stackable.

Data model

Two Firestore collections. All relationship data is denormalised into the cat document to minimise read operations.

Field Type Notes
name string Cat's primary name
alternativeNames string[] Other names the cat is known by
emoji string Colour-coded emoji used as map marker (8 types)
location CatLocation lat, lng, area, city, country, continent, approximateAddress — all resolved from pin coordinates via Nominatim reverse geocode at creation time
photos CatPhoto[] url, contributorId, date, objectPosition (for focus picker)
descriptions Description[] Community text: id, text, type, contributor, date
visits Visit[] userId, userName (first name only), date (ISO string)
userVisits Record<uid, count> Denormalised per-user visit count for fast profile lookups
totalVisits number Maintained via Firestore increment()
slowBlinks SlowBlink[] userId, userName, date
contributors Contributor[] id, name, type (creator / photo / info), contributions count
creatorId / creator string Original author. Set to "anonymous-user" / "Anonymous user" when anonymised

The users collection mirrors stats back onto the profile document (catsFound, photosAdded, totalContributions, catsVisited[]) for display without requiring cross-collection queries on every profile view.

05

Safety architecture

Location data about living animals carries genuine risk. Safety was treated as a product feature, not a compliance checkbox. Each layer addresses a specific threat model.

Implemented

Login wall

Full auth required before any data is accessible. No public-facing cat map. Firestore rules enforce request.auth != null on all reads.

Implemented

Location blur

Cat profile maps randomise the pin ±500m and pull zoom back 3 levels. Neighbourhood context retained, precise address obscured.

Implemented

Street-level trust gate

Approximate street address hidden from accounts under 30 days old with fewer than 3 contributions. Visible text: "Street detail visible after 30 days or 3 contributions."

Implemented

Visit rate limiting

Visits limited to once per cat per 24 hours. Prevents systematic "visit every cat" data collection patterns. Enforced client-side with server timestamp comparison.

Implemented

First names only

firstName() utility applied at both display time and write time. Full names never stored in cat-related documents. Email prefix fallback replaced with "Catwalker".

Implemented

Anti-theft statement

Visible on the login screen before any interaction. Framing: "Catwalk celebrates cats in your community — it is not a tool for harm."

Implemented

Duplicate submission guard

A submittingRef boolean blocks reentry on the cat creation form. Prevents accidental duplicate cats from rapid taps during photo upload.

Implemented

Firestore auth rules

Cats collection reads require request.auth != null. Prevents unauthenticated scraping of location data via the Firebase API.

Next iteration

Report / flag system

Allow users to flag a cat profile or user as suspicious. Firestore field + admin review queue. Not yet implemented.

Next iteration

Zoom-level marker gating

Hide cat pins until user zooms past a threshold, preventing "pull back and map the whole neighbourhood" reconnaissance from the map view.

07

Key design decisions

Decision Alternatives considered Rationale
Single App.tsx File-per-component structure Enabled rapid iteration with AI tools. Full context always visible. Can be refactored when team scales.
Login required for everything Public browse, login for contribution Safety: unauthenticated users cannot scrape cat locations via API. Community: all data is attributed.
Denormalised Firestore model Normalised collections with joins Eliminates N+1 reads for the map view. Cat document is the source of truth for all its activity.
First names only, always Display names stored verbatim Privacy by default. Applied at write time so the choice is permanent, not just a display toggle.
No external UI library MUI, shadcn, Radix Full visual control. Faster initial load. Forces intentional design rather than default component behaviour.
Granular contribution deletion Delete by type only GDPR-adjacent right to erasure. User should control exactly what's removed — specific photos, specific visits, specific descriptions — not face an all-or-nothing choice.
Nominatim for geocoding Google Maps API, Mapbox Free, open-source, no API key required for low-volume use. Consistent with the OSM map tiles.
submittingRef not useState for submit guard Disable button via state A ref is synchronous — it blocks reentry immediately, before React re-renders. State-based guards can still race if the user taps twice in the same frame.
firstName() applied at write time Apply only at display time Display-only privacy is fragile — if the display logic changes, the data is exposed. Writing first name only to Firestore makes the privacy choice permanent and unconditional.

Build process

Catwalk was built entirely through conversational AI-assisted development — from the first component to the production deployment — without a local development environment.

The workflow was: describe a feature or problem in plain language → receive code → paste into GitHub → Vercel auto-deploys → test on device → iterate. The entire development cycle from idea to deployed feature routinely took under ten minutes.

This project is a proof-of-concept for a different kind of product development: one where the barrier between product thinking and production code is close to zero. The bottleneck is no longer "can this be built" — it's "is this worth building, and is it safe."

Version control was managed via GitHub's browser interface and the macOS Terminal. No IDE, no local npm, no build tools installed on the machine. The entire codebase — 7,497 lines of TypeScript — was written, debugged, and refactored through dialogue, including TypeScript error resolution, React state bugs, Firestore rule changes, mobile layout fixes, and real user-reported issues from live testing.

The most instructive moments were the bugs: a hardcoded "London" for cat location that only surfaced when a user in Perth added a cat; a duplicate submission problem caused by a user tapping the button twice during photo upload; an email prefix showing in the visit list instead of a name. Each was a product decision that had been deferred, and each had a clean fix once the actual data was in front of us. Real usage found things that no amount of speculative planning would have.

The process revealed which product decisions are genuinely complex (safety architecture, data model, user trust levels, privacy at write-time vs display-time) and which only appeared complex because of the traditional gap between design and engineering. That gap — and the months it normally takes to close it — was largely absent here.

Feature concept: cat-relevant locations

The natural extension of a cat map is a map of the infrastructure cats depend on — and that their communities use. A second data layer of cat-relevant locations would sit alongside the cat pins, togglable as an overlay, and serve both practical and community-building functions.

Core idea: cats exist in an ecosystem. Knowing where Puss lives is more useful if you also know the nearest vet is two streets away, the pet shop stocks her favourite food, and there's a cat café on the same road. The location layer connects the cats to their world.

Location types

🏥 Veterinary practices

Standard and 24-hour emergency vets. Opening hours, phone number, distance from current position. Surfaced prominently in a future "emergency vet finder" flow.

🛍 Pet shops

Community-submitted independent and chain pet shops. Users can tag which cats in the area they stock food for, or note if the shop knows local cats.

☕ Cat cafés

Cat café locations with resident cat profiles linked directly to their Catwalk entries. Bridges the community map to formal cat-hospitality businesses.

🍽 Feeding stations

Community feeding points for strays and semi-ferals. Sensitive location type — visible only to trusted users (same 30-day / 3-contribution trust gate as street addresses).

🌿 Cat-friendly gardens

Parks, community gardens, or private spaces whose owners have indicated cats are welcome. Useful for catwalkers exploring a new neighbourhood.

🏠 Cat rescue centres

Local rescue organisations and foster networks. Could integrate with adoption listings in a later version.

Data model extension

A new locations Firestore collection, separate from cats, with a similar contributor model. Key fields:

Product considerations

Design principle

Overlay, not replacement

Location pins appear as a toggleable layer — they don't clutter the cat map by default. Users opt in to seeing them.

Design principle

Community-maintained, not scraped

Locations are submitted by users, not imported from Google Places or Yelp. This maintains the community-knowledge character of the product and avoids API costs.

Safety consideration

Feeding stations

Feeding station locations require the same trust gate as cat street addresses — visible only to accounts 30+ days old or with 3+ contributions. Protects stray colonies from disturbance.

Future integration

Emergency vet finder

A dedicated "find nearest open vet" button in the app header — filters the locations layer for 24hr practices, sorted by distance from current position. High-value, low-engineering-cost feature once the data layer exists.

Monetisation

Catwalk's monetisation should feel like a natural extension of the product — not a tax on community participation. Core features (the map, cat profiles, visiting, contributing) stay free. Revenue comes from users and businesses who want more, or who want to be more visible.

Principle: nothing that generates revenue should compromise the safety architecture, the community character, or the data privacy of the people and animals in the product.

Catwalk Pro Subscription

A small monthly or annual subscription for power users. Unlocks: full personal visit history export, advanced analytics on cats you've added (visit trends, slow blink counts over time), early access to new features, and a Pro badge on your public profile. Critically, no core functionality is paywalled — Pro is additive, not restrictive. Target price: £2.99/month or £24.99/year.

£2.99/mo
individual
Verified business listings B2B subscription

Once the cat-relevant locations layer is built (Section 10), businesses — vets, pet shops, cat cafés, rescue centres — can claim and verify their listing. A verified listing gets: a badge, rich profile (services, opening hours, photos, staff bios), priority map visibility, and the ability to link their location to cats in the area. Community-submitted locations remain free and unverified. Target price: £15–35/month depending on location type and tier. Low churn category — a vet practice's listing is evergreen value.

per business
Data insights B2B / institutional

Anonymised, aggregated neighbourhood cat density data has genuine value to third parties — not individual cat data, but area-level insights: which neighbourhoods have the highest stray density, seasonal presence patterns, concentration around specific location types. Potential buyers: local councils managing stray populations, vet groups doing catchment analysis, animal welfare NGOs doing urban cat research, pet food brands doing geographic targeting. Delivered as periodic reports or API access. Requires a data processing agreement and transparent user disclosure (already architecturally sound — no PII in the cat documents). Target: £500–5,000/report depending on scope.

Visual design — intent

The current visual design is functional but provisional. Emoji markers and the default purple/blue colour palette are placeholders that allowed the product to be built and tested quickly. The intended visual direction is substantially more considered.

The current version proves what it needs to prove: the product works, the data model is sound, the safety architecture holds, and people use it. The next phase is to make it feel like something you'd want to show someone.

Planned

Custom cat illustrations

Replace emoji markers on the map and browse cards with a set of hand-drawn cat illustrations — one per colour type (black, grey, orange, white, tabby, etc.). Distinctive, ownable, and immediately readable at small map sizes. The illustrations become the brand.

Planned

Cohesive colour system

The current hyperlink blue and grey palette was chosen for legibility and neutrality during build. The intended palette is warmer — developed around the product's character rather than inherited from a default. Cat profiles could carry a subtle tint derived from the cat's colour type.

Planned

Shape language

Cards, modals, and buttons currently use generic border radii. The redesign would develop a shape vocabulary specific to Catwalk — soft but not childish, expressive but not chaotic. Profile cards and map markers would feel like they belong to the same visual family.

Planned

Map marker redesign

Replacing the emoji div-icon markers with SVG illustrations per cat colour type. Custom marker shadow and hover states. Cluster treatment for dense areas that resolves back to illustrations on zoom. The map should feel like it was made for cats specifically, not borrowed from a generic mapping template.
